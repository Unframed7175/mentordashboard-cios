  <name>Task 1: Create utils/klassen.js — multi-class state management</name>
  <files>utils/klassen.js</files>
  <action>
Create `utils/klassen.js` as a classic script (no ES module) exposing window.* globals.

**Storage key:** `var KLASSEN_KEY = 'mentordashboard_klassen_v1';`

**State object:**
```javascript
window.klassenState = {
  klassen: {},          // { [klasId]: { id, naam, students, lastSaved } }
  activeKlasId: null,
};
```

**Functions to implement (all as `window.*`):**

1. `window.createKlas(naam)` — per D-01/D-03/D-04:
   - Generate ID: `'klas_' + Date.now().toString(36)`
   - Create class object: `{ id, naam, students: [] }`
   - Add to `klassenState.klassen[id]`
   - Call `switchActiveKlas(id)` to activate
   - Call `saveKlassen()`
   - Return the new klas object
   - Duplicate name guard: if any existing class has the same name (case-insensitive), return `{ error: 'duplicate', naam: naam }` instead of creating. This is a soft guard per research — not a hard constraint, but prevents accidents.

2. `window.switchActiveKlas(klasId)` — per D-07, research Pattern 2:
   - Validate `klasId` exists in `klassenState.klassen`; return false if not
   - Set `klassenState.activeKlasId = klasId`
   - **CRITICAL bridge:** `window.appState.students = klassenState.klassen[klasId].students` — same array reference, so addStudent/mergeVerzuim mutate the right array
   - Call `saveKlassen()`
   - Return true

3. `window.deleteKlas(klasId)` — per D-09/KLS-05:
   - Remove `klassenState.klassen[klasId]`
   - If deleted class was active: switch to first remaining class, or set `activeKlasId = null` and `window.appState.students = []` if no classes remain
   - Call `saveKlassen()`
   - Return true

4. `window.saveKlassen()` — per KLS-04:
   - Serialize `{ klassen: klassenState.klassen, activeKlasId: klassenState.activeKlasId, savedAt: new Date().toISOString() }` to `localStorage.setItem(KLASSEN_KEY, ...)`
   - Wrap in try/catch; return boolean
   - Also call `window.saveState()` for backward compat (harmless dual-write)

5. `window.loadKlassen()` — per KLS-04/KLS-06:
   - Try `localStorage.getItem(KLASSEN_KEY)`
   - If found: parse JSON, restore `klassenState.klassen` and `klassenState.activeKlasId`, establish the bridge (`window.appState.students = active class students`)
   - Return `true` if at least one class exists
   - If NOT found: call `window._migrateV1ToKlassen()` (migration)
   - Return false if no data at all

6. `window._migrateV1ToKlassen()` — v1.0 auto-migration:
   - Check `localStorage.getItem('mentordashboard_v1')`
   - If data found with students array: create one class named "Klas 1", populate with old students, save to klassen key, remove old key
   - Establish bridge. Log: `[klassen.js] Migratie van v1 naar multi-klas uitgevoerd (N leerlingen -> Klas 1)`
   - Return true if migration succeeded, false otherwise
   - Notes key (`mentordashboard_notities`) does NOT need migration — leerlingIds are stable

7. `window.getActiveStudents()` — convenience getter:
   - Return active class's students array, or empty array if no active class

**Anti-patterns to avoid (from research):**
- Do NOT namespace `mentordashboard_leerlijnen_v1` per class (D-08)
- Do NOT call `window.clearState()` inside `deleteKlas()` — it orphans the array reference. Operate on `klassenState` directly.
- Do NOT set `window.appState.students = []` anywhere except in `switchActiveKlas()` and the no-classes-remain fallback in `deleteKlas()`

Add `console.log('[klassen.js] Multi-class manager loaded');` at end.
