<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Mentor → number input | Free-form input — could be decimal, negative, out-of-range, non-numeric (NaN from `Number("")`) |
| Settings UI → utils/normen.ts saveNormen | Goes through clamp + round before persistence |
| utils/normen.ts → store.json (plugin-store) | Tauri IPC, atomic write per LazyStore guarantees |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-25-07 | Tampering | Mentor input — decimal / negative / huge values | mitigate | handleNormenBlur applies `Math.round()` and clamps to per-field min/max (D-09) before saveNormen. Acceptance criterion verifies Math.round usage. Manual checkpoint steps 7+8 confirm. |
| T-25-08 | Denial of Service | Empty / non-numeric input causes NaN | mitigate | `Number.isFinite(rawValue) ? rawValue : DEFAULT_NORMEN[field]` fallback in handleNormenBlur ensures NaN never reaches saveNormen. ASVS V5 Input Validation. |
| T-25-09 | Spoofing / Information Disclosure | onNormenChanged firing during partial typing leaks intermediate value | accept | Save is gated on blur/Enter only (D-06). onChange updates local state only — never calls saveNormen. No external observer of intermediate state. |
| T-25-10 | Tampering | Stale React closure in handleNormenBlur captures old `normen` | mitigate | Pattern uses `setNormen(updated)` + spreads `{ ...normen, [field]: clamped }` from the latest render's closure. Because saveNormen is awaited in the handler and the handler is called synchronously from onBlur, the closure value is fresh. Manual checkpoint step 4 (multi-edit + restart) detects any stale-write regression. |

No high-severity threats. ASVS V5 Input Validation satisfied at the UI boundary; persistence boundary covered by Plan 02 threat model.
</threat_model>
