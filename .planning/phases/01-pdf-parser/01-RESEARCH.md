## Summary

Phase 1 builds the PDF parser for voortgang-PDFs from CIOS Zuidwest-NL. The PDFs are generated by the school system and follow a consistent structure, but the parsing is non-trivial because PDF.js extracts raw text items with coordinates rather than structured data — and the central challenge is reconstructing the 19-column "Overzicht Deelgebieden" table (which may span 3 pages) from raw positional text.

The existing sister project (`get-shit-done-main/dashboard`) already solved the same domain problem for Excel format, giving us verified: the 19 deelgebied IDs and labels (in `utils/schema.js`), the `normalizeScore()` function, and the in-memory data model structure. The PDF parser must produce the same data shape. The key difference is that Excel has a grid structure that SheetJS resolves; PDFs require manual coordinate-based reconstruction.

PDF.js version 5.5.207 (latest as of 2026-03-24) is the correct library. It is available as vendored ESM files from jsDelivr. The main build is `pdf.min.mjs` (~415 KB) and the worker is `pdf.worker.min.mjs` (~1.2 MB) — larger than ideal but unavoidable. Processing 19 PDFs sequentially (not parallel) avoids memory pressure. The table reconstruction algorithm clusters text items by Y-coordinate proximity, then resolves column assignments by comparing X-position of each cell against the stored X-positions of the header row.

**Primary recommendation:** Use PDF.js 5.5.207 vendored ESM with position-based table reconstruction. Reuse `DEELGEBIEDEN` and `normalizeScore()` from the existing schema.js unchanged. Process PDFs sequentially to manage memory, update counter after each completion.

### Pattern 1: PDF.js ESM Browser Setup
**What:** Import PDF.js as an ES module, set workerSrc to the vendored worker file.
**When to use:** Every time; this is the required setup for browser usage without a build step.
**Example:**
```javascript
// parsers/pdf.js
// Source: https://cdn.jsdelivr.net/npm/pdfjs-dist@5.5.207/build/
import * as pdfjsLib from '../vendor/pdf.min.mjs';

// CRITICAL: workerSrc must be set before first getDocument() call
// Use a relative URL resolved from the current page
pdfjsLib.GlobalWorkerOptions.workerSrc = new URL(
  '../vendor/pdf.worker.min.mjs',
  import.meta.url
).href;
```

**File:// protocol note:** PDF.js workers are disabled for `file://` URLs. The app requires a local HTTP server (e.g., `start.bat` with Python `http.server` or similar). This is consistent with the existing dashboard's `start.bat` pattern.

### Pattern 3: PDF Text Item Extraction via getTextContent()
**What:** Extract all text items with their X/Y positions from each page.
**When to use:** Core of the parser; all parsing is built on this.
**Example:**
```javascript
async function extractPages(arrayBuffer) {
  const pdf = await pdfjsLib.getDocument({ data: arrayBuffer }).promise;
  const allItems = [];

  for (let pageNum = 1; pageNum <= pdf.numPages; pageNum++) {
    const page = await pdf.getPage(pageNum);
    const content = await page.getTextContent();

    // Each item: { str, transform: [a,b,c,d,x,y], width, height, hasEOL }
    for (const item of content.items) {
      if (!item.str || item.str.trim() === '') continue;
      allItems.push({
        str:    item.str,
        x:      item.transform[4],   // X position in PDF coordinate space
        y:      item.transform[5],   // Y position (bottom-up; higher = higher on page)
        width:  item.width,
        height: item.height,
        page:   pageNum,
      });
    }
    page.cleanup(); // free canvas resources
  }
  return allItems;
}
```

**Coordinate system:** PDF coordinate origin is bottom-left; Y increases upward. Within a page, higher Y means higher on the page. Across pages, page 2 items have separate Y values — treat each page independently or offset by page number for cross-page sorting.

### Pattern 4: Header Field Extraction (Naam, Leerling ID, Periode)
**What:** Find key-value pairs in the page header using regex matching against sorted text.
**When to use:** PDF-02 (header fields).
**Example:**
```javascript
function extractHeader(items) {
  // Sort items by page, then descending Y (top to bottom)
  const sorted = [...items].sort((a, b) =>
    a.page !== b.page ? a.page - b.page : b.y - a.y
  );

  const header = {};
  const textLines = groupIntoLines(sorted, Y_TOLERANCE); // see Pattern 5

  for (const line of textLines.slice(0, 20)) { // header is in first ~20 lines
    const text = line.map(i => i.str).join(' ').trim();
    const naam    = text.match(/^Naam\s*:\s*(.+)/i);
    const leerlingId = text.match(/^Leerling\s*ID\s*:\s*(.+)/i);
    const periode = text.match(/^Periode\s*:\s*(.+)/i);
    if (naam)       header.naam = naam[1].trim();
    if (leerlingId) header.leerlingId = leerlingId[1].trim();
    if (periode)    header.periode = periode[1].trim();
  }

  return header;
}
```

### Pattern 5: Y-Position Line Grouping (Core Algorithm)
**What:** Cluster text items into logical rows using Y-coordinate proximity tolerance.
**When to use:** All table/line reconstruction; foundation of PDF-03, PDF-04, PDF-05.
**Example:**
```javascript
const Y_TOLERANCE = 3; // points; items within 3pt of each other = same line

function groupIntoLines(items, tolerance = Y_TOLERANCE) {
  // Items should be sorted by page asc, Y desc (top to bottom), X asc
  const lines = [];
  let currentLine = [];
  let currentY = null;

  for (const item of items) {
    if (currentY === null || Math.abs(item.y - currentY) <= tolerance) {
      currentLine.push(item);
      // Maintain weighted average Y to handle slight baseline shifts
      currentY = currentY === null ? item.y
        : (currentY * (currentLine.length - 1) + item.y) / currentLine.length;
    } else {
      if (currentLine.length) {
        // Sort items within line by X (left to right)
        lines.push(currentLine.sort((a, b) => a.x - b.x));
      }
      currentLine = [item];
      currentY = item.y;
    }
  }
  if (currentLine.length) lines.push(currentLine.sort((a, b) => a.x - b.x));
  return lines;
}
```

**Calibration note:** Y_TOLERANCE of 3 is a reasonable starting point for typical 11-12pt body text. The table header row in "Overzicht Deelgebieden" uses abbreviated column labels (V&A, M&M, etc.) that may render at a slightly smaller font size — test against actual PDFs and adjust if needed.

### Pattern 6: Deelgebied Table Column Detection
**What:** Find the header row of "Overzicht Deelgebieden" and record X-position of each column.
**When to use:** PDF-05 (the hardest requirement).
**Example:**
```javascript
const DEELGEBIED_LABELS = ['V&A','M&M','INS','O&DW','C&B','1E&B','P&O','S&O',
                           'ORG','I&B','2E&B','PrCo','VSK','LOB','INFO','DESK',
                           'BS','TOW','BH'];
const COLUMN_X_TOLERANCE = 8; // points; column assignment tolerance

function findDeelgebiedHeaderRow(lines) {
  for (const line of lines) {
    const texts = line.map(i => i.str.trim().toUpperCase());
    // Match if ≥5 deelgebied labels found in the same line
    const matchCount = DEELGEBIED_LABELS.filter(
      lbl => texts.includes(lbl.toUpperCase())
    ).length;
    if (matchCount >= 5) return line; // found the header row
  }
  return null;
}

function buildColumnMap(headerLine) {
  // Returns: { 'V&A': xPosition, 'M&M': xPosition, ... }
  const map = {};
  for (const item of headerLine) {
    const upper = item.str.trim().toUpperCase();
    const dg = DEELGEBIED_LABELS.find(l => l.toUpperCase() === upper);
    if (dg) map[dg] = item.x;
  }
  return map;
}

function assignScoreToColumn(item, columnMap) {
  // Find the nearest column header X within tolerance
  let best = null, bestDist = Infinity;
  for (const [label, colX] of Object.entries(columnMap)) {
    const dist = Math.abs(item.x - colX);
    if (dist < bestDist && dist <= COLUMN_X_TOLERANCE) {
      bestDist = dist;
      best = label;
    }
  }
  return best; // null if no column matched
}
```

**Critical observation:** The "Overzicht Deelgebieden" table may repeat its header row on each page it spans. The parser must handle this by re-using the same `columnMap` (column X positions should be consistent across pages for the same PDF) and not creating duplicate rows.

### Pattern 7: Feed Forward and Status Extraction
**What:** Within each vak section, extract per-opdracht status and feed forward text.
**When to use:** PDF-03, PDF-04.
**Example:**
```javascript
const STATUS_STRINGS = [
  'Op tijd ingeleverd en wel beoordeeld',
  'Zelfevaluatie afgerond',
];

function parseVakSection(lines, vakName) {
  const opdrachten = [];
  let currentOpdracht = null;

  for (const line of lines) {
    const text = line.map(i => i.str).join(' ').trim();

    // Opdracht header: typically starts with a number or dash + opdracht name
    if (looksLikeOpdrachHeader(text)) {
      if (currentOpdracht) opdrachten.push(currentOpdracht);
      currentOpdracht = { naam: text, status: '', feedForward: '' };
      continue;
    }

    if (!currentOpdracht) continue;

    // Status detection
    const matchedStatus = STATUS_STRINGS.find(s =>
      text.toLowerCase().includes(s.toLowerCase())
    );
    if (matchedStatus) {
      currentOpdracht.status = matchedStatus;
      continue;
    }

    // Feed Forward label triggers capture of following lines
    if (/feed\s*forward/i.test(text)) {
      currentOpdracht.capturingFeedForward = true;
      continue;
    }
    if (currentOpdracht.capturingFeedForward && text) {
      currentOpdracht.feedForward += (currentOpdracht.feedForward ? ' ' : '') + text;
    }
  }
  if (currentOpdracht) opdrachten.push(currentOpdracht);
  return opdrachten;
}
```

### Anti-Patterns to Avoid
- **Concatenating all text without coordinates:** `items.map(i => i.str).join(' ')` destroys spatial structure needed for table reconstruction. Always preserve X/Y.
- **Strict Y equality for row matching:** PDF rendering introduces sub-pixel baseline shifts. Always use a tolerance window (Y_TOLERANCE).
- **Sorting items globally across pages:** Page 1 Y-coordinates overlap with page 2 Y-coordinates. Group by page first, sort within page.
- **Assuming column order in raw items:** PDF.js does NOT guarantee items are returned left-to-right within a line. Always sort items by X within each line group.
- **Promise.all for 19 PDFs simultaneously:** Each getDocument() call spins up a Web Worker and loads ~1.2 MB. Running 19 in parallel can cause memory pressure and worker pool exhaustion. Use sequential or batched processing instead.
- **Using CDN at runtime (no offline support):** The requirement is browser-only with no external dependencies — vendor PDF.js files locally.

## Common Pitfalls

### Pitfall 1: Multi-Page Table — Repeated Header Rows
**What goes wrong:** "Overzicht Deelgebieden" spans up to 3 pages. On each new page, the table header row repeats. If the parser treats each page independently, it will try to parse the header row as data, or duplicate the column map, causing corrupted scores.
**Why it happens:** PDF.js returns all items per page; the parser must correlate items across pages to recognize table continuation.
**How to avoid:** Detect the header row pattern (line containing ≥5 deelgebied abbreviations). Once found, store the `columnMap`. On subsequent pages, detect the same pattern and skip (re-use existing columnMap rather than overwriting). The X positions of columns should be identical across pages in the same PDF.
**Warning signs:** Duplicate opdrachten in deelgebied table; missing scores for leerlingen on page 2+.

### Pitfall 2: Y-Coordinate Direction (Bottom-Up)
**What goes wrong:** PDF coordinate system has Y=0 at the bottom. Text near the top of the page has HIGHER Y values than text at the bottom. If you sort by Y ascending, you get text bottom-to-top, which reverses the document order.
**Why it happens:** PDF spec defines origin at bottom-left; this is opposite to screen coordinates.
**How to avoid:** Sort by Y descending (`b.y - a.y`) to get top-to-bottom order within a page.
**Warning signs:** Last section appears first; header fields parsed from footer area.

### Pitfall 3: Items Not in Left-to-Right Order
**What goes wrong:** PDF.js does not guarantee text items within a visual line are returned left-to-right. In table rows, score items may arrive in arbitrary order.
**Why it happens:** PDF rendering order is determined by the PDF's drawing commands, not visual position.
**How to avoid:** Always sort items within a grouped line by X ascending before reading column values.
**Warning signs:** V appears where G should be; deelgebied scores systematically wrong for some columns.

### Pitfall 4: Worker Disabled on file:// Protocol
**What goes wrong:** PDF.js Web Worker fails silently on `file://` URLs. The library will attempt a fallback to main-thread parsing, which may work partially but is unreliable and slow.
**Why it happens:** Browser security policy prevents Web Workers from loading scripts via `file://`.
**How to avoid:** Always serve via HTTP (the existing `start.bat` does this). Document this requirement clearly.
**Warning signs:** Parsing works for small PDFs but hangs or fails for larger ones.

### Pitfall 5: Score Values Not Just V/G/E
**What goes wrong:** The parser assumes only V, G, E appear in the deelgebied table. In practice, some cells may contain dashes (—), dots, or the letter O (Onvoldoende), or may be empty.
**Why it happens:** Different assessment stages may produce different output; early in a period some deelgebieden have no score yet.
**How to avoid:** Use the existing `normalizeScore()` which already handles O/V/G/E + full Dutch words + returns null for unrecognized values. null means "empty/not assessed".
**Warning signs:** Parser errors on specific students; deelgebied counts wrong for prognosis.

### Pitfall 6: Feed Forward Text Fragmentation
**What goes wrong:** Feed forward text may be long (multiple sentences) and is split across many `item.str` values by PDF.js, sometimes with unexpected whitespace.
**Why it happens:** PDF.js splits text at font changes, hyphenation points, and spacing boundaries.
**How to avoid:** Concatenate all text items belonging to the same "line" group, then concatenate consecutive lines that are part of the feed forward paragraph until the next structural marker (new opdracht, new vak heading, or "Overzicht Deelgebieden" section).
**Warning signs:** Feed forward text truncated; only first sentence captured.

## Recommended Data Model

The output of the PDF parser (in-memory, Phase 1 only) should match the data shape already used by the Excel parser in the sister project, making Phase 2 integration straightforward.

```javascript
// Single student parsed from one PDF:
{
  // From PDF header (D-01 to D-02)
  naam:       "Bosker, J.G. (Javier-Andrès)",
  leerlingId: "12345",
  periode:    "BJ2 Fase 2 DD",
  leerjaar:   "2",
  filename:   "Voortgang-5-BJ2 Fase 2 DD-Bosker, J.G. (Javier-Andrès).pdf",

  // Per-vak opdracht feedback (D-03 to D-04)
  vakken: [
    {
      naam: "Sport en Beweging",
      opdrachten: [
        {
          naam:        "Opdracht 1: Lesontwerp",
          status:      "Op tijd ingeleverd en wel beoordeeld",
          feedForward: "Goed gedaan, let op de structuur van de les.",
        },
        // ...
      ]
    }
  ],

  // Deelgebied scores (D-10, D-14, PDF-05)
  // Key = deelgebied label (matches DEELGEBIEDEN[*].label)
  // Value = normalizeScore() result: 'voldoende' | 'goed' | 'excellent' | null
  deelgebiedScores: {
    'V&A':  'goed',
    'M&M':  'voldoende',
    'INS':  null,         // not yet assessed
    'O&DW': 'excellent',
    // ... all 19 deelgebieden
  },

  // Per-opdracht deelgebied scores (for Phase 3 gap analysis)
  // Array of { vakNaam, opdrachNaam, scores: { 'V&A': 'goed', ... } }
  datapunten: [
    {
      vak:       "Sport en Beweging",
      datapunt:  "Opdracht 1: Lesontwerp",
      scores:    { 'V&A': 'goed', 'M&M': 'voldoende' }
    }
  ]
}

// Full app state after import:
window.parsedStudents = []; // StudentRecord[]
```

**Key decisions for data model:**
- `deelgebiedScores` is a flat map aggregating the "best" score per deelgebied across all datapunten — suitable for doorstroomnorm prognosis (Phase 3 NORM-03/04/05).
- `datapunten` preserves the raw per-opdracht scores, enabling future gap analysis (Phase 5 DET-04).
- Use the same deelgebied label keys as `DEELGEBIEDEN[*].label` to enable direct lookup without transformation.

| PDF.js UMD global (`pdfjs-dist/build/pdf.js`) | PDF.js ESM (`pdf.min.mjs`) | v3.x → v4.x (2023) | Must use `import` or `<script type="module">`, not `<script src>` with globals |
| `pdfjsLib.getDocument(url)` | `pdfjsLib.getDocument({ data: arrayBuffer })` | Stable since v2 | ArrayBuffer approach avoids CORS issues; works with File API offline |
| `FileReader.readAsArrayBuffer()` | `file.arrayBuffer()` (Promise) | ES2020 | Cleaner async code; supported in all modern browsers |
| normalizeWhitespace option in getTextContent() | Option removed/ignored since ~v3 | ~2022 | Whitespace is now always normalized; don't rely on this option |

**Deprecated/outdated:**
- `pdfjsLib.workerSrc = '...'` (direct property): Replaced by `pdfjsLib.GlobalWorkerOptions.workerSrc`.
- `page.getTextContent({ normalizeWhitespace: false })`: The option is now ignored; PDF.js normalizes whitespace by default.
- Importing from `pdfjs-dist/build/pdf.js` (CommonJS UMD): The modern build is `.mjs` ES modules only.

## Open Questions

1. **Exact Y_TOLERANCE value for this PDF template**
   - What we know: Tolerance of 3 points is standard for body text in 11-12pt. The "Overzicht Deelgebieden" header row uses abbreviations that may render at a smaller font size.
   - What's unclear: The exact rendered font size and baseline variation in these CIOS PDFs. May need 2-4 for header rows and 3-5 for body text.
   - Recommendation: Start with Y_TOLERANCE = 3 for all rows. Add a debug mode (console.log of grouped lines) to calibrate against real PDFs before finalizing.

2. **Feed Forward text encoding**
   - What we know: Feed forward is free text entered by teachers. May contain special characters, Dutch diacritics (é, ë, ü), em-dashes.
   - What's unclear: Whether PDF.js faithfully reproduces these characters or loses encoding on some fonts.
   - Recommendation: Test with PDFs that contain diacritics. PDF.js generally handles UTF-8 and standard Western fonts; most encoding issues appear with specialized fonts.

3. **Section boundary detection between vakken**
   - What we know: PDFs are structured by vak (subject). The boundary between sections is likely a larger-font heading or a visually distinct line.
   - What's unclear: The exact visual marker PDF.js will produce (larger font size changes `transform[0]` scaling; bold may appear as a different fontName).
   - Recommendation: After the first test parse, use the font size (derived from `transform[0]`) to identify headings. `transform[0]` equals the font size in points for upright text.
