<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → GitHub Release | GITHUB_TOKEN used to create release and upload assets; write permission explicitly declared |
| Internet → end-user machine | Unsigned binary downloaded over HTTPS; no additional checksum verification for internal use |
| macOS Keychain → app (first launch) | Ad-hoc signed binary triggers one-time user dialog to allow keychain access |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15-04 | Tampering | GitHub Release artifacts | accept | Artifacts served over HTTPS from GitHub. No additional checksum needed for internal school tool. SHA-256 checksums can be added to release notes in a future phase if required. |
| T-15-05 | Information Disclosure | release.yml secrets | mitigate | Only GITHUB_TOKEN (auto-provided, write-scoped to this repo only) and APPLE_SIGNING_IDENTITY (literal "-", not a secret) are in the workflow. No cert private keys or Apple credentials embedded. Acceptance criterion explicitly checks TAURI_SIGNING_PRIVATE_KEY is absent. |
| T-15-06 | Elevation of Privilege | NSIS installer on Windows | mitigate | installMode "currentUser" (set in Plan 01) ensures no UAC elevation. Verified by acceptance criterion in Plan 01 Task 2. |
| T-15-07 | Spoofing | Unsigned macOS binary | accept | Ad-hoc signing embeds a hash of the binary — not a verified developer identity, but sufficient to pass Apple Silicon's code-signing requirement. Teachers are instructed in D-05 to use Privacy & Security > "Open Anyway" for first launch. Acceptable for internal use. |
</threat_model>
