<objective>
Fix BUG-01: files dragged onto the import dropzone are silently ignored because Tauri 2's built-in OS drag-drop handler intercepts the events before the HTML5 DataTransfer API fires, leaving `e.dataTransfer.files` empty.

Purpose: Restore working drag-and-drop file import for PDF, .xls, and .zip files.

Output: One-line config change in tauri.conf.json that disables Tauri's interceptor, plus a version bump in Cargo.toml to align the package version with the current milestone.
</objective>

<task type="auto">
  <name>Task 1: Disable Tauri's built-in drag-drop interceptor</name>
  <files>src-tauri/tauri.conf.json</files>
  <action>
In `src-tauri/tauri.conf.json`, add `"dragDropEnabled": false` as the last property inside `app.windows[0]`, immediately after the existing `"useHttpsScheme": true` line.

The result for that window object must be:

  {
    "title": "Mentordashboard CIOS",
    "width": 1200,
    "height": 800,
    "resizable": true,
    "fullscreen": false,
    "useHttpsScheme": true,
    "dragDropEnabled": false
  }

Root cause context: Tauri 2 registers a native OS-level drag-drop handler on every window by default. This handler consumes the OS drag event before the browser's HTML5 DataTransfer pipeline runs, so `e.dataTransfer.files` is always an empty FileList. Setting `dragDropEnabled: false` in the window config tells Tauri to skip its native handler and pass the raw OS events through to the WebView, restoring normal HTML5 drag-drop behaviour.

CRITICAL — do NOT touch `src/App.tsx`. The document-level `dragover` and `drop` addEventListener calls with `preventDefault()` that exist there are a separate, intentional safeguard that prevents browser navigation when a user accidentally drops a file outside the designated dropzone. They are not related to this bug.

After saving the file, a full Tauri dev server restart (`npm run tauri dev`) is required for the config change to take effect — the hot-reload watcher does not pick up tauri.conf.json changes.
  </action>

  </verify>
  <done>"dragDropEnabled": false is present inside app.windows[0]; dragging a PDF/xls/zip onto the dropzone populates the file list; dropping outside the dropzone does not navigate the browser.</done>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| OS → WebView | File paths arrive via the OS drag event; Tauri passes them to the browser after dragDropEnabled is disabled |
| Browser → App logic | HTML5 DataTransfer.files carries File objects from the OS; the dropzone handler reads them |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-20-01 | Spoofing | dragDropEnabled: false | accept | The change only disables Tauri's native intercept; it does not open new trust paths. File content is still validated by the existing import pipeline (file type checks on extension and MIME type). |
| T-20-02 | Tampering | document-level drop listener | accept | The existing preventDefault() guard in App.tsx is preserved verbatim; it prevents arbitrary URL navigation from drag-drop outside the dropzone. No new attack surface is introduced. |
| T-20-03 | Information Disclosure | File path via DataTransfer | accept | File objects are scoped to the renderer process and processed in-memory by the import pipeline. No file paths are persisted or transmitted externally. |
</threat_model>
