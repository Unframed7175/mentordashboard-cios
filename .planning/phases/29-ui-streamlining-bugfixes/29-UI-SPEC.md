## Typography

All text uses the Industry font family. Two weights only: regular (400) and bold (700).

| Role | Size | Weight | Line Height | Token / Class |
|------|------|--------|-------------|---------------|
| Section title | 0.6875rem (11px) | 700 | 1.2 | `.detail-section-title` — uppercase, `var(--text-faint)` |
| Body / content text | 0.875rem (14px) | 400 | 1.5 | Default `<p>` and data cells |
| Label / badge | 0.75rem (12px) | 700 | 1.3 | Criterion row labels, status chips, score-telling |
| Card name | 1rem (16px) | 700 | 1.2 | Leerlingnaam in `.klas-tile` and detail header |

**Canonical rules for UI-01 typography consistency pass:**

- All section titles across all views (klasoverzicht, detailweergave, settings, help) use exactly: `font-size: 0.6875rem; font-weight: 700; text-transform: uppercase; color: var(--text-faint); letter-spacing: 0.06em`.
- Body text in every section: `font-size: 0.875rem; font-weight: 400; color: var(--text-secondary)`.
- Empty state and muted hint text: `font-size: 0.875rem; font-weight: 400; color: var(--text-muted)`.
- Status badges / score chips: `font-size: 0.75rem; font-weight: 700`.
- No other font sizes are introduced in this phase.

Source: CONTEXT.md "Established Patterns — Section titles" and RESEARCH.md Open Questions §2 recommendation.

## Color

All color values must come from CSS custom properties. No hardcoded hex is permitted in `.tsx` component files in this phase. Violations catalogued in RESEARCH.md "UI-03 Dark Mode Audit" table must all be resolved.

| Role | Light value | Dark override | Usage |
|------|-------------|---------------|-------|
| Dominant surface (60%) | `var(--bg-page)` = `#F1F5F9` | `#0B0F1A` | Page background, outer wrappers |
| Secondary surface (30%) | `var(--bg-surface)` = `#FFFFFF` | `#131929` | Cards, tiles, sections, nav |
| Accent (10%) | `var(--accent)` = `#009FE3` | `#009FE3` (same) | See reserved list below |
| Destructive | `var(--status-rood-text)` = `#991B1B` | `#FCA5A5` | Error messages, delete confirm text only |

**Accent (`#009FE3`) reserved for:**
- Active nav tab bottom border indicator
- Focus ring on interactive inputs
- `.nav-stripe` gradient (FIX-01 replacement div)
- Primary action buttons (e.g. "Importeren", "Opslaan")
- Settings gear icon active state
- Spider chart `prof_handelen` stroke color (existing, unchanged)

**RAG color system (3-state criterion rows — PROG-01):**

| State | Background token | Text token | Usage |

| Groen | `var(--status-groen-bg)` | `var(--status-groen-text)` | Criterion met: `nodig === 0` |

## Component Contracts

### DoortstroomPrognoseSection — new block layout (PROG-01)

**Layout:** Three blocks stacked vertically. On viewports ≥ 900px, SBL + SBC may lay side by side (`display: flex; flex-wrap: wrap; gap: 1rem`), with Negatief block below full-width.

**BJ1 variant:** Show "BJ2 doorstroom" block + "Versneld SBC" block + Negatief block. Detect via `status.prognose.traject === 'bj1'`.

**Block anatomy:**

```
┌─────────────────────────────────────────────┐
│ [block-header]  BLOCK NAME     [status chip] │
│ ─────────────────────────────────────────── │
│   [criterion row]  Label · score / thr  [•]  │
│   [criterion row]  Label · score / thr  [•]  │
└─────────────────────────────────────────────┘
```

**Block header:** CSS class `.prognose-block-header`. Contains block name as text (`font-size: 0.75rem; font-weight: 700; text-transform: uppercase; color: var(--text-faint)`) and an overall status chip using `.status-groen` / `.status-oranje` / `.status-rood`.

**Block container:** CSS class `.prognose-block`. Border: `1px solid var(--border-default)`. Radius: `var(--radius-md)`. Background: `var(--bg-surface-alt)`. Left border accent: `3px solid` using the block's overall status color (`var(--rag-groen)` / `var(--rag-oranje)` / `var(--rag-rood)`).

**Criterion row:** CSS class `.prognose-criterion-row`. Layout: `display: flex; justify-content: space-between; align-items: center; padding: 4px 0`. Contains:
- Left: label text (`font-size: 0.875rem; color: var(--text-secondary)`)
- Center: `score / threshold` display (`font-size: 0.875rem; font-weight: 700; color: var(--text-primary)`)
- Right: status dot or small chip using `.status-groen` / `.status-oranje` / `.status-rood`

**Empty state (no scores):** Each block body shows `"Nog geen scores beschikbaar"` in `var(--text-muted)` at `0.875rem`. The block header still renders with the block name; no status chip is shown.

**Data mapping (BJ2):**

| Block | Criterion | Score field | Threshold field | Status rule |
|-------|-----------|-------------|-----------------|-------------|
| SBL | "≥{n.sbl} deelgebieden ≥V" | `totaalVoldoendeOfHoger` / `n.sbl` | `n.sbl` | groen: `gaps.nodigSBL===0`, oranje: `<=2`, rood: `>2` |
| SBC | "≥{n.sbc} deelgebieden ≥V" | `totaalVoldoendeOfHoger` / `n.sbc` | `n.sbc` | groen: `gaps.nodigSBC_deelgebieden===0`, oranje: `1–2`, rood: `>2` |
| SBC | "Kerndeelgebieden ≥V" | `(4 - gaps.nodigSBC_kern.length)` / 4 | 4 | groen: kern.length===0, rood: otherwise |
| Negatief | "≤{n.negatiefTotaal} O totaal" | `totaalOnvoldoende` vs `n.negatiefTotaal` | `n.negatiefTotaal` | groen: `!isNegatief`, rood: `isNegatief` |
| Negatief | "≤{n.negatiefPerLeerlijn} O per leerlijn" (×3) | per-leerlijn `onvoldoende` | `n.negatiefPerLeerlijn` | groen: `onvoldoendeRuimtePerLeerlijn[l] >= 0`, rood: negative |

Thresholds come from `getNormenSync()` — never hardcoded.

Source: CONTEXT.md D-PROG-01..05, RESEARCH.md "New DoortstroomPrognoseSection Block Layout".

### Nav stripe (FIX-01)

Remove `#main-nav::after` CSS block entirely. Add `<div aria-hidden="true" className="nav-stripe" />` as last child inside `<nav id="main-nav">` in `KlasTabStrip.tsx`.

CSS for `.nav-stripe`:
```css
.nav-stripe {
  position: absolute;
  top: 0;
  right: 0;
  width: 140px;
  height: 52px;
  background: linear-gradient(to bottom-left, #009FE3 0%, #009FE3 25%, transparent 70%);
```

Note: The gradient uses the brand accent `#009FE3` directly (not `var(--accent)`). This is acceptable for this purely decorative element as the gradient stops must be exact color literals for predictable rendering in WebView2.

## Interaction Contracts

### View transitions (UI-04)

All three transition targets use `var(--transition-base)` = `200ms ease`.

| Target | Mechanism | Implementation |
|--------|-----------|----------------|
| Klasoverzicht ↔ DetailWeergave | Fade + 4px vertical slide | Add `.view-fade-in` class to top-level wrapper div of both `KlasOverzicht` and `DetailWeergave`. CSS animation fires on every mount. |
| Tab navigation active indicator | Color + border-bottom animate | `transition: all var(--transition-base)` already on `.nav-tab` — verify and confirm no regression |
| Settings panel open/close | Slide from right | `.view-slide-in-right` + `@keyframes slideInFromRight` already in `index.css` section 27 — verify and complete |

`.view-fade-in` CSS:
```css
.view-fade-in {
  animation: viewFadeIn var(--transition-base) ease forwards;
}
@keyframes viewFadeIn {
  from { opacity: 0; transform: translateY(4px); }
  to   { opacity: 1; transform: translateY(0); }
}
```

All transitions must respect `@media (prefers-reduced-motion: reduce)` — verify the existing `prefers-reduced-motion` block in `index.css` covers these new animation classes and add them if not.

Existing hover patterns (from Phase 19) are preserved unchanged. No new hover interactions are introduced in this phase.
